Sql的编译与执行-parser

SQL 编译与执行系列技术博客将按照以下顺序分别介绍整个 SQL 执行引擎。

SQL 编译与执行研读流程

  • parser 部分，包括词法解析和语法解析。

  • compile 部分，包括语义解析以及计划的构建。

  • optimize 部分，包括计划的优化。

  • exec 部分，包括执行计划的生成以及执行。

一、SQL 执行流程

图二所示为一条 SQL 语句的整体处理流程，总体来说，一条 SQL 语句需要经过下述步骤：parser—构建逻辑计划—构建优化计划—构建物理计划—构建执行计划。

parser 过程最主要的目的就是解析输入的 SQL 语句，通过词法解析器解析为 token，通过语法解析器生成抽象语法树，而后即可传入 SQL 执行引擎进行识别和处理。

接着进入下一步，首先要对输入的数据进行有效性验证，解析并转换 AST 树，构建逻辑计划。接下来就是对生成的计划进行优化以找到代价最小的执行方式，根据优化好的逻辑计划构建可执行的物理计划，最后下发到各个节点进行分布式执行。

二、Lex & Yacc

Lex 代表 Lexical Analyzar（词法分析器），Yacc 代表 Yet Another Compiler Compiler（编译器代码生成器）。它们分别是用来生成词法分析器和语法分析器的工具，Lex 和 Yacc 在 UNIX 下分别叫 Flex 和 Bison。

词法解析是编译的第一步，将语句拆分为 token，移除空格和注释，逐个读入源码中的 character，检查是否有效并传递给语法分析器。语法分析器以 token-stream 的形式从词法分析器获取输入；解析器根据规则文件生成的规则将 token 解析成一个抽象语法树的结构，如图三所示。

从上图的执行流程可以看出，我们需要分别为 Lex 和 Yacc 提供对应的规则文件，Lex & Yacc 根据给定的规则，生成符合需求的词法分析器和语法分析器。一般这两种配置都是文本文件且结构相同：

文件通过 %% 分成三部分，最上面定义了各种名称，例如各种表达式、token 等，中间则是重点的规则

  • 定义部分：这部分包含各种名称的定义，例如各种表达式、token等。
  • 规则部分：这部分包含了所有的词法规则。每一条规则通常由一个正则表达式和一个动作组成。当输入文本与正则表达式匹配时，就会执行相应的动作。
  • 子程序部分：这部分包含了被规则部分引用的各种辅助函数。

首先看一下 Lex 的规则文件：

对于词法解析对应的规则，可以看出，左边是扫出来的字符内容，通过正则表达式进行匹配，如果匹配到即返回右边大括号中运行的结果。

每一行代表一条规则，每条规则由正则表达式和相应的行为组成。

  1. [a-z]：当输入字符匹配小写字母时（满足 [a-z]），将字符转换为对应的整数值（例如 ‘a’ 转换为 0，’b’ 转换为 1，等等），并且将返回值设为 VARIABLE（一种token类型）。
  2. [0-9]+：当输入匹配一个或多个数字（满足 [0-9]+）时，将输入的字符串转换为整数值，并将返回值设为 INTEGER（另一种token类型）。
  3. [-+()=/*\n]：当输入匹配任一运算符（包括 +，-，(，)，=，/，*，或者换行符）时，返回该符号的 ASCII 值。
  4. [ \t]：当输入字符为空格或者制表符时（满足 [ \t]），忽略它并进行下一个字符的处理。
  5. .：对于所有其他的输入，调用 yyerror 函数并传入错误消息 “invalid character”，表示输入了一个无法识别的字符。

在上述规则中，yylval 是 Lex 和 Yacc 之间通信的一个全局变量。当 Lex 识别出一个 token 时，它将相关的值存储在 yylval 中，然后 Yacc 可以通过这个变量取得这个值。

再看看 Yacc 对应的规则文件：

  1. %token INTEGER VARIABLE：这定义了两个 token 类型，即 INTEGER 和 VARIABLE，这些类型是由 Lex 词法分析器生成的。
  2. %left '+' '-' 和 %left '*' '/'：这定义了运算符的结合性和优先级。这里，’+’ 和 ‘-‘ 具有相同的优先级，而 ‘*’ 和 ‘/‘ 有更高的优先级。所有这些运算符都是左结合的，即当优先级相同时，先进行左边的运算。
  3. program:：定义了 program 的语法规则。一个 program 可以是一个 program 后面跟着一个 statement 和一个换行符，或者也可以是空。
  4. statement:：定义了 statement 的语法规则。一个 statement 可以是一个 expr（表达式），或者是一个 VARIABLE 后面跟着一个等号和一个 expr。
  5. expr:：定义了 expr 的语法规则。一个 expr 可以是一个 INTEGER，或者是一个 VARIABLE，或者是两个 expr 通过 ‘+’, ‘-‘, ‘*’, ‘/‘ 运算符连接，或者是一个在括号内的 expr。

在每个规则的右边，大括号 {} 内的是该规则的动作代码，它通常是一段 C 代码。例如，在规则 expr: expr '+' expr { $$ = $1 + $3; } 中，当 parser 匹配到这个规则时，它将执行 { $$ = $1 + $3; }，也就是将两个表达式的值相加，然后将结果赋值给当前表达式。

$1, $2, $3 这种表示方式代表的是规则右边的元素，$1 是第一个元素，$2 是第二个元素，以此类推。而 $$ 则是规则左边的元素，也就是当前正在构造的元素。

语法规则使用了巴科斯范式（BNF）定义，它不仅能严格地表示语法规则，而且所描述的语法是与上下文无关的。它具有语法简单，表示明确，便于语法分析和编译的特点。每条规则的左部是一个非终结符，右部是由非终结符和终结符组成的一个符号串。具有相同左部的规则可以共用一个左部，各右部之间以直竖“|”隔开。

解析表达式是生成表达式的逆向操作，我们需要归宿表达式到一个非终结符。Yacc 生成的语法分析器使用自底向上的归约（shift-reduce）方式进行语法解析，同时使用堆栈保存中间状态。简单的以一条 select 为例：

SELECT * FROM TEST 为例

让我们以SQL查询 “SELECT * FROM test” 为例，来解释词法分析和语法分析的过程，以及Lex和Yacc之间的通信过程。

首先，这个SQL查询会被送到词法分析器（在这里就是Lex）。Lex会把这个输入字符串转化为一系列的tokens。Lex中定义的规则会用于匹配输入的字符或字符串，然后转换成相应的token。在这个例子中，”SELECT”, “ * “, “FROM”, 和 “test” 分别被识别为SELECT关键字、特殊字符’*’、FROM关键字和一个VARIABLE（假设我们的规则定义表名为VARIABLE）。

这些tokens会连同它们的值（如果有）一起传递给Yacc进行语法分析。例如，”SELECT” 和 “FROM” token可能就直接表示为它们自己，而VARIABLE “test” 的值可能就是它的名称 “test”。

三、常用的SQL parser

https://hackernoon.com/14-open-source-sql-parsers

以下是一些主要的开源解析器：

  1. MySQL/MariaDB：

    Pingcap parser是用Go编写的MySQL解析器。

    SQL Parser in phpmyadmin是一个专注于MySQL方言的验证SQL词法分析器和解析器。

  2. Postgres：libpg_query从Postgres项目中提取解析器（用C编写）并将其打包为独立的库。这个库被其他项目用其他语言包装，如

    Python的pglast，

    Ruby的pg_query，

    Golang的pg_query_go，

    JS的psql-parser和pg-query-emscripten

    Rust的pg_query.rs。

  3. 多引擎：

    queryparser实现了Apache Hive，Presto/Trino和Vertica方言。

    zetasql实现了BigQuery，Spanner和Dataflow方言。

  4. 通用解析器：

    Python的sqlparse

    Rust的sqlparser-rs

    Python的mo-sql-parsing

  5. 平台：解析器/优化器平台实现了通用的SQL语言特性，并允许定制作为平台的一等特性。两个流行的开源项目是Apache Calcite和JSQLParser。

    Apache Calcite是一个流行的解析器/优化器，被Apache Hive，BlazingSQL等流行的数据库和查询引擎使用。

    JSQLParser可以解析多种SQL方言，如MySQL，Postgres和Oracle。语法可以修改以支持其他SQL方言。

如果您的数据库没有解析器怎么办？

大多数团队不会从头开始创建解析器。一个流行的选择是使用 Postgres 解析器，然后添加自定义 SQL 语法。AWS Redshift、Vertica 和 DuckDB 就是示例。使用 Postgres SQL 解析器解析这些数据库的查询历史记录，以解析大多数查询。

ANTLR（Another Tool for Language Recognition）是一个强大的解析器生成器，它可以构建和遍历解析树，用于从结构化文本数据中提取信息。然而，ANTLR本身并不是一个特定的SQL解析器，而是一个工具，可以用来创建各种语言的解析器，包括SQL。

在上述文章中，ANTLR并没有被直接列为一个SQL解析器，但是它被提到作为一个创建解析器的工具。例如，有些开源项目可能使用ANTLR来创建特定于MySQL或Postgres等数据库的SQL解析器。

所以，ANTLR本身并不直接算作14个开源SQL解析器之一，但它是创建这些解析器的重要工具之一。

Python的sqlparse库可以用来解析SQL语句，但它并不直接提供抽象语法树（AST）。只能通过遍历sqlparse的解析结果来获取类似AST的结构。

在Python的sqlparse库中，每个token都是一个Token对象，它有一个ttype属性和一个value属性。ttype属性表示token的类型，value属性表示token的值。

在这个输出中，你可以看到每个token的Python类型（例如，sqlparse.sql.Token或sqlparse.sql.Identifier），ttype（例如，Token.Keyword.DML或Token.Text.Whitespace），以及value（例如，SELECT或 ）。

然而，如果你需要完整的AST，你可能需要使用更强大的解析工具，如ANTLR。ANTLR可以生成完整的AST，并提供遍历和操作AST的工具。你可以为ANTLR提供SQL的语法规则（通常为.g4文件），然后使用ANTLR生成Python的解析器代码。然后，你可以在Python代码中使用这个解析器来解析SQL语句并生成AST。这通常需要对ANTLR和解析技术有更深入的了解。
