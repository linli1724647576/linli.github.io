History-Driven Build Failure Fixing: How Far Are We?(2019)

Basic Information:

  • Title: History-Driven Build Failure Fixing: How Far Are We? (历史驱动的构建失败修复：我们走了多远？)
  • Authors: Yiling Lou, Junjie Chen, Lingming Zhang, Dan Hao, Lu Zhang
  • Affiliation: HCST (Peking University), China

论文简要 :

  • 本研究通过构建一个新的大型构建失败数据集，对历史驱动的构建失败修复技术进行了评估，并提出了一种基于当前信息的简化修复技术。实验结果表明，基于当前信息的方法成功修复了比基于历史信息的方法多2倍的可重现构建失败。

背景信息:

  • 论文背景: 构建系统在现代软件开发中起着重要作用，但是构建系统在软件演化过程中经常出现故障。因此，自动化构建失败修复技术需求巨大。
  • 过去方案: 过去的研究提出了各种不同类型的构建失败修复技术，其中包括基于历史信息的修复技术。然而，这些方法的有效性和适用性仍存在一定的限制。
  • 论文的Motivation: 本文旨在评估历史驱动的构建失败修复技术的有效性和局限性，并提出一种基于当前信息的简化修复技术。通过对比实验结果，揭示了当前信息在自动化构建失败修复中的潜力和未来研究的指导意义。

方法:

  • a. 理论背景:
    • 本文介绍了构建系统在软件开发中的重要性以及构建失败的频繁发生。它提到了关于构建失败修复技术的现有研究，并介绍了HireBuild，这是一个用于Gradle的基于历史的构建失败修复工具。然而，它也强调了HireBuild的局限性，并指出了对构建失败修复技术的进一步研究和改进的需求。
  • b. 技术路线:
    • HoBuFF使用两步方法来修复软件项目中的构建失败。在故障定位步骤中，HoBuFF搜索分配给同一配置元素的值。它可以在项目内部搜索值，也可以从资源（如Gradle中央仓库、Gradle DSL文档和Android DSL文档）中搜索值。外部资源提前收集并以结构化形式表示。在修补生成步骤中，HoBuFF基于定位的根因语句生成修补候选。它应用不同的修补操作符，如插入、更新和删除，生成修补候选。然后，验证生成的修补程序以确定其正确性。

结果:

  • a. 详细的实验设置:
    • HoBuFF使用Groovy AST API实现，并在具有特定硬件和软件规格的计算机上进行评估。实验针对一组可重现的构建失败进行，对每个失败应用HoBuFF生成修补程序。根据成功的构建任务完成和匹配的编译文件大小等标准对修补程序进行验证。
  • b. 详细的实验结果:
    • HoBuFF成功修复了102个可重现的构建失败中的18个（18%），而最先进的工具HireBuild只修复了9个（9%）。在HireBuild修复的9个构建失败中，HoBuFF也修复了8个。这表明在修复构建失败方面，HoBuFF的表现优于HireBuild。HoBuFF在修复与内部元素相关的失败方面特别有效，其中内部配置元素的值对项目是特定的。但在修复与外部元素相关的失败方面，HoBuFF的效果较差，这些失败是由于外部配置元素的不正确值引起的。

ABSTRACT

这篇论文的主题是自动化构建修复技术，尤其是针对广泛应用于开源Java项目的主流构建系统Gradle。近期提出的最先进的针对Gradle的构建修复工具HireBuild，通过挖掘历史修复记录来修复Gradle构建失败。虽然已经证明HireBuild在修复实际Gradle构建失败方面非常有效，但是其在有限的构建失败集上进行了评估，并且在很大程度上依赖于历史修复信息的质量和可用性。

为了调查历史驱动构建修复的有效性和局限性，作者首先构建了一个从GitHub的Top-1000项目中收集的新的大型构建失败数据集。然后，他们在这个扩展数据集上定量和定性地评估了HireBuild。受研究发现的启发，他们提出了一种简单的新技术，该技术通过从当前测试项目和外部资源中搜索，而不是历史修复信息，来生成可能的补丁。

根据他们的实验结果，基于现有信息的简单方法比基于历史修复信息的最新技术HireBuild成功地修复了2倍的可重现构建失败。此外，他们的结果还揭示了未来高级构建失败修复的各种发现/指南。

KEYWORDS 自动程序修复，构建系统，构建故障修复

INTRODUCTION

这篇论文的背景是现代软件开发中广泛使用的构建系统（例如Gradle，Ant和Maven）及其对应的构建脚本。这些构建脚本在软件演进过程中频繁更新，以保持与改变的源代码或环境（例如，第三方库和插件）的一致性。如果出现不一致或错误，构建脚本可能会引发构建失败。构建失败在商业和开源软件系统中都频繁发生，并且可能严重推迟软件开发中的其他活动。因此，广泛的构建失败问题已经引起了软件工程研究者的越来越多的关注，对不同类型的构建失败进行了各种研究/技术。

相关工作中，例如，HireBuild是一种在ICSE’18提出的最先进的通用构建失败修复技术，它从历史项目中成功的修复中学习修复模式，并通过应用学习到的修复模式生成补丁。但是，HireBuild的有效性仅在有限的数据集上进行了评估，且作为一种基于历史的技术，其有效性依赖于训练数据的质量和可用性。

该论文指出了HireBuild的局限性，并通过构建一个新的、大型的实际构建失败数据集，对HireBuild进行了全面的评估，以充分了解基于历史的构建修复技术HireBuild的效用和限制，并推动未来的构建修复研究。他们对数据集中可复现的构建失败进行了HireBuild的性能复查，也对HireBuild的成功和失败案例进行了定性手动检查。

根据这项研究的发现，作者提出了一种轻量级的构建失败修复技术——HoBuFF（不依赖历史的构建失败修复），它不依赖历史数据，而只是简单地利用构建代码、构建日志和外部构建相关资源的当前信息。HoBuFF包括两个阶段：（1）故障定位，（2）补丁生成。在进行HireBuild和HoBuFF的实证比较后，他们发现在102个可复现的失败中，HoBuFF能够在更短的时间内成功修复18个错误，包括HireBuild修复的8个错误（HireBuild总共修复了9个错误）。

这篇论文的贡献包括：

  • 数据集：包含375个实际构建失败中的102个可复现构建失败的数据集，这比现有的构建失败数据集要大得多，并可作为未来构建修复研究的基准数据集。
  • 研究：对扩展数据集中最先进的基于历史的构建失败修复（HireBuild）进行了广泛研究，并对其优点和局限性进行了详细的手动检查。
  • 技术：一个新的构建失败修复技术（HoBuFF），只需要当前信息（即，无需历史数据），通过使用轻量级的数据流分析和查询内部/外部资源来进行构建修复。
  • 启示：对HoBuFF和最先进的HireBuild进行的实证评估，表明当前项目信息可以大大补充历史构建修复信息，用于自动化构建失败修复，并揭示了未来高级构建失败修复的各种发现/指南。

BACKGROUND

2.1 Build Failure Fixing:Challenges

作者通过一个名为Mockito/db8a3f3的构建失败和其手动修复补丁的示例，说明了修复Gradle构建失败的挑战。

在这个示例中，表1展示了一个构建失败示例，包含从构建日志中获取的相关错误信息片段（即错误消息）和其手动修复补丁。在执行任务releaseNeeded期间，构建过程在第104行终止，因为第104行试图访问属性ext.needed，而这个属性在else分支上并没有被定义。为了修复这个失败，开发人员在第102行后面添加了一个额外的语句ext.needed = false。

在含有与失败相关信息的构建日志给出的情况下，第一个挑战在于在Gradle脚本中定位故障。如表1中的错误消息所示，第104行是触发构建失败并停止构建过程的代码位置。然而，根据手动补丁，修复是在第102行之后添加的。第104行揭示了构建失败，但它可能并非构建失败的根本原因。因此，如果没有识别出所有可能的根本原因语句，我们可能无法修复构建失败。

即使已经确定了根本原因语句的集合，还存在另一个挑战：如何生成正确的补丁。具体来说，要修复这个示例中的构建失败，自动化修复技术需要找出错误属性ext.needed的正确值，这需要理解程序。在其他情况下，当构建失败是由于使用了外部资源（例如，第三方库依赖）的错误值引起的，自动化修复技术也需要对外部资源有开放的了解。[8]

在Gradle构建系统中，一个“任务（Task）”是构建过程中要执行的一个步骤。任务可以执行各种不同的操作，如编译Java源代码、创建一个JAR文件、生成Javadoc等。每个任务都由一系列的操作（Action）组成，这些操作在执行任务时按顺序执行。

“releaseNeeded”是一个特定的任务名称，可能特定于某个项目或构建过程。这个任务的具体作用取决于它在Gradle构建脚本中的定义。根据这个任务的名称，它可能是用来检查是否需要进行软件发布的任务，但这需要查看具体的项目和构建脚本来进行确认。

在论文中的这个例子中，”releaseNeeded”任务在执行过程中出现了错误，因为它试图访问一个在某个执行路径上未被定义的属性ext.needed，这导致了构建失败。

2.2 HireBuild

HireBuild 是一个现有的自动构建失败修复技术。它的工作方式主要分为两个阶段：模式提取（Pattern Extraction）和补丁生成（Patch Generation）。

在模式提取阶段，HireBuild 首先需要一个训练集，该训练集由历史上收集的构建修复组成。然后，它从训练集中选择一些错误信息与给定构建失败的错误信息相似的构建修复，将这些选定的构建修复视为种子修复（Seed Fixes）。HireBuild 从这些种子修复中提取模式，并使用一些启发式策略对它们进行排序。

在补丁生成阶段，HireBuild为四种类型的构建元素定义了一些规则，并在这些元素的模式中填入具体的值。这四种元素类型包括：（1）标识符（包括任务名、块名、变量名），（2）Gradle 插件和第三方工具/库的名称，（3）项目内的文件路径，和（4）版本号。然后，HireBuild会根据补丁和种子修复之间的相似度对生成的补丁进行排序，将补丁应用到匹配模式的代码位置，并最后按顺序验证它们。

然而，HireBuild虽然考虑了这四种类型的元素，但仍有一些其他元素（如项目特定变量）的具体值在生成补丁时是必需的，但HireBuild并未考虑。对于这些元素，HireBuild并没有为生成具体值设计特定的规则，因此不能为相应的构建失败生成候选补丁。这就是HireBuild的一些局限性。

对于表1中的示例 ext.needed（whose value is true or false）不属于这四种类型中的任何一种。

STUDY ON HIREBUILD

在关于HireBuild的研究中，研究人员发现HireBuild在非常小的数据集上的评估可能导致过拟合，因此有必要在一个不同且更大的数据集上重新评估其性能。研究还强调了手动检查HireBuild成功和失败案例的必要性，以了解其效能和可能的限制。

为了减少由于在原始HireBuild研究中使用小数据集所带来的有效性威胁，研究人员扩展了现有的数据集。他们从GitHub上最受欢迎的1000个Java项目中收集了额外的375个构建失败（其中102个可重现）。新收集的数据集被称为”扩展数据集”。

本研究提出的研究问题围绕HireBuild在扩展数据集上的表现，其修复构建错误的成功或失败的原因，以及其限制。研究发现，HireBuild只修复了扩展数据集中的9%的构建失败，这个成功率比在其原始数据集中报告的成功率（46%）低得多，暗示可能存在过拟合问题。

论文提供了一些关于自动修复构建失败的指导原则：

  1. 使用历史数据的必要性：历史修复数据并不是构建失败修复技术的必需因素。相反，更重要的是充分利用当前信息（如当前的脚本代码，当前的构建日志和内部/外部资源）。
  2. 使用当前数据的可行性：对于给定的构建失败，通常可以获得当前的脚本代码和构建日志。此外，许多程序分析技术可以用来分析脚本代码。再者，构建日志通常结构良好，允许直接提取错误信息。对于资源，有结构良好的官方文档和存储库可供自动参考。
  3. 分析更多的构建代码模式：由HireBuild提取的模式较为死板，因为它保留的程序信息很少。这暗示，从构建代码中分析更多的修复模式可以帮助获取修复构建失败的关键线索。
  4. 考虑构建代码中的更多元素：在生成补丁时考虑的构建代码中的元素类型有限，这也限制了HireBuild的效能。这暗示，一个更通用/系统的构建失败修复方法也需要考虑更多的脚本元素。

A NEW TECHNIQUE: HOBUFF

在对HireBuild研究的发现的启发下，我们提出了一种轻量级的构建失败修复技术，名为HoBuFF（忽视历史的构建失败修复），它不以历史修复为输入，而是以更全面的方式利用当前信息。HoBuFF主要包含两个阶段：基于数据流的故障定位和基于搜索的补丁生成。在故障定位的第一阶段，HoBuFF通过分析构建日志提取错误信息，然后通过应用轻量级的数据流分析定位可能的错误代码。在补丁生成的第二阶段，HoBuFF设计修复操作符并搜索修复成分，以生成补丁候选项。为了方便理解，我们将在整个章节中使用第2节中提出的例子来说明我们的方法。

4.1 基于数据流的故障定位

当构建失败时，构建日志会记录相应的错误信息，这对手动定位构建失败的根源非常有帮助。因此，HoBuFF进行故障定位的第一步是从构建日志中提取错误信息。基于提取的错误信息，HoBuFF然后定位显示构建过程中构建失败的语句，这就是在构建脚本中暴露构建失败的语句。最后，HoBuFF通过轻量级的跨过程数据流分析，从显示错误的语句中追溯到错误的根源。

4.1.1 错误信息提取。构建日志可能记录了许多与构建过程的各个阶段相关的信息，比如初始化和任务执行等。因此，HoBuFF首先解析构建日志，提取与构建失败相关的信息，也就是错误信息。由于Gradle构建日志的标准形式，日志中存在错误指示头来标记错误信息。HoBuFF使用这些错误指示头从构建日志中提取错误信息。此外，由于构建失败的原因可能完全不同，设计固定的提取模板来提取所有错误元素名称是不现实的，因此我们采取了不同的策略，包括POS标记、异常名词提取、特殊标记处理等，以获取潜在的错误元素名称和值进行进一步分析。

总的来说，将会收集以下特定的错误信息来帮助故障定位过程：(1) 项目，(2) 任务，(3) 配置元素，(4) 元素的值，(5) 显示错误的语句的位置。

4.1.2 显示错误的语句识别。显示错误的语句可能并不是失败的根本原因，但实际上它对于识别根本原因非常重要。如果存在最后一种信息，也就是显示错误的语句的位置，HoBuFF可以根据行号直接在构建脚本中找到显示错误的语句。否则，HoBuFF会使用其他类型的信息推断构建脚本中的显示错误的语句。

4.1.3 根源定位。为了在构建脚本中识别根源，HoBuFF从显示错误的语句反向追溯，执行跨过程的数据流分析。请注意，我们只考虑数据流依赖关系（忽略控制流依赖关系），以避免过度近似，使得分析更为轻量级。首先，HoBuFF构建一个跨过程的控制流图，并用通过可达定义分析计算出的数据依赖信息进行标注。然后，基于图，HoBuFF识别所有影响显示错误语句（包括显示错误的语句）的语句，作为潜在的根源语句，这就是故障定位过程的输出。

4.2 基于搜索的补丁程序生成

本章节主要介绍了如何使用基于搜索的方法生成补丁，来修复构建失败的问题。这一章分为三个部分，分别是修复运算符（4.2.1），修复成分（4.2.2）和补丁候选者生成（4.2.3）。

  1. 修复运算符：HoBuFF定义了三种修复运算符，包括更新、插入和删除，这些运算符主要用于修改buggy的配置。
    • 更新: 通过替换错误的值ψb为正确的值ψ+b，来更新配置cb。
    • 插入: 插入一个配置cb，其中eb是通过错误定位确定的，ψ+b是需要引入的成分。
    • 删除: 删除配置cb，不需要任何成分。为了避免语法问题，进一步分析从cb到删除所有受影响的语句的数据依赖关系。
  2. 修复成分：为了修复错误，HoBuFF需要查找正确的修复成分。根据它们的来源，这些成分可以分为内部和外部两种类型。
    • 内部搜索: HoBuFF会在项目内部搜索所有赋值给相同配置元素的值。
    • 外部搜索: HoBuFF会从外部资源中搜索值。这些外部资源包括Gradle中心仓库、Gradle DSL文档和Android DSL文档。
  3. 补丁候选者生成：补丁候选者生成过程的输入是一组定位的根源语句，输出是构建失败的补丁候选者列表。具体地说，补丁候选者生成过程可以分为三个层次：首先，对于每个根源语句，HoBuFF首先决定应用哪种修复运算符，这取决于语句是否为空定义；然后，对于每种修复运算符，HoBuFF直接生成一个补丁候选者，如果是”删除”，而如果是”插入”或”更新”，HoBuFF通过内部或外部搜索生成修复成分；最后，对于每个修复成分，HoBuFF生成一个补丁并验证它。

假设你正在使用Gradle构建一个Android项目，并遇到了一个构建失败的问题。构建日志显示失败的原因是“找不到com.example:mylib:1.0.0这个依赖”。

在这种情况下，HoBuFF将会启动以下步骤：

  1. 错误定位（Fault Localization）：HoBuFF首先会分析构建日志和脚本，确定问题的原因是Gradle脚本中的一个依赖声明（com.example:mylib:1.0.0）有误。
  2. 修复操作符的选择：因为这个错误的依赖声明是一个错误的值（也就是说，这个依赖版本并不存在），所以HoBuFF会按照”更新”-“插入”-“删除”的顺序选择修复操作符。
    • 首先，”更新”操作符会被应用，HoBuFF将尝试替换错误的依赖版本号。
  3. 修复成分（Fixing Ingredient）的生成：为了找到正确的依赖版本，HoBuFF会在Gradle中心仓库中搜索com.example:mylib这个库，并找到一个存在的版本，比如1.0.1。
  4. 生成并应用补丁：HoBuFF将会生成一个包含新依赖版本的补丁，也就是将com.example:mylib:1.0.0替换为com.example:mylib:1.0.1，并应用到原始的Gradle脚本上。
  5. 补丁验证：应用了新补丁的Gradle脚本会被重新执行，如果构建成功，那么这个补丁就被视为有效的补丁。

这就是一个HoBuFF修复Gradle构建错误的具体例子。

EMPIRICAL EVALUATION

两个主要研究问题是：

• RQ4：HoBuFF在修复构建失败的数量上的表现如何？

• RQ5：HoBuFF在修复构建失败所需时间和生成候选修复方案的数量上的表现如何？

研究使用Groovy AST API实现了HoBuFF，并在一台配备64个Intel(R) Xeon e5 CPU核心、128GB内存和Ubuntu 14.04.1的计算机上进行了所有实验。

在RQ4的评估中，HoBuFF成功修复了102个可复现构建失败中的18个（18%），而最新的HireBuild只修复了其中的9个（9%）。这表明了仅使用当前项目信息，而非使用历史修复信息进行构建失败修复的优势。此外，HireBuild修复的9个构建失败中，有8个也被HoBuFF修复了。

在RQ5的评估中，研究发现HoBuFF在成功修复失败后的平均修复时间为156秒，显示出其在修复真实世界构建失败中的效率。在两种技术成功修复失败的情况下，他们都没有花费很长的时间（即少于800秒）。但是，HoBuFF的修复时间集中在较小的范围内（即少于200秒），比HireBuild更优秀。

然而，尽管HoBuFF在很大程度上超越了最新的HireBuild，但它并没有修复所有的构建失败。在这部分，研究调查了不能被我们的方法修复的剩余构建失败，以了解其在故障定位和修复方案生成方面的局限性。研究发现，无法被HoBuFF修复的84个构建失败中，有58个是由于HoBuFF故障定位阶段的局限性，而另外26个是由于修复方案生成阶段的局限性。
