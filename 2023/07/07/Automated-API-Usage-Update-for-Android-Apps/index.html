Automated API-Usage Update for Android Apps

Basic Information:

  • Title: Automated API-Usage Update for Android Apps
  • Authors: Mattia Fazzini、Qi Xin、Alessandro Orso
  • Affiliation: Georgia Institute of Technology Atlanta, GA, USA
  • Keywords: Mobile apps, automated update, API analysis
  • ISSTA 2019

摘要

移动应用程序高度依赖其底层操作系统（OS）提供的应用程序编程接口（API）。由于操作系统和API可能会频繁变化，开发者必须快速更新他们的应用程序，以确保应用程序在新的API和操作系统版本下能够按预期运行。为了帮助开发者处理这个繁琐、容易出错且耗时的任务，我们开发了一种技术，可以根据其他开发者如何针对同样的变化升级他们的应用程序的示例，自动执行应用程序的更新以适应API的变化。给定一个需要更新的目标应用程序和API变化的信息，我们的技术执行四个主要步骤。首先，它分析目标应用程序，以确定受API变化影响的代码。其次，它搜索现有的代码库，寻找更新到新版本API的示例。第三，它分析、排名，并将在上一步中找到的更新示例转化为通用的补丁。最后，它按照排名顺序将生成的补丁应用到目标应用程序上，同时执行差异测试以验证更新。我们实现了我们的技术，并对15个真实世界的应用程序进行了实证评估，结果令人鼓舞。总的来说，我们的技术能够更新85%的考虑的API变化，并自动验证68%的执行的更新。

INTRODUCTION

背景

本文的背景是移动应用程序在很大程度上依赖于其底层操作系统（OS）和应用程序编程接口（API）。然而，操作系统和API经常更新，这就需要开发者及时更新他们的应用程序以适应新的API和操作系统版本。这个过程往往是繁琐、容易出错和耗时的，尤其是当应用程序需要在最新的操作系统和早期版本上都能运行时。此外，虽然API的变化通常在API文档中有所讨论，但在大多数情况下，这些文档并未提供如何更新使用改变的API的软件的相关示例。因此，开发者必须自己找出如何更新他们的代码。由于应用程序中API的使用通常非常广泛，因此这些更新可能会非常大。

Motivation

本篇工作的直觉是，通过利用现有的代码库，即通过查找和分析已经经历了同样更新的现有应用程序的代码，可以在很大程度上自动化地将一个应用程序从其底层API的旧版本演变到新版本。基于这个直觉，作者们开发了AppEvolve。

相关工作

Although the problem of performing code updates based on examples has been studied before, most existing techniques (e.g., [8,39, 52]) mainly target repetitive updates within the same code base.Moreover, these techniques typically assume that update examples are provided as inputs.

相关的研究工作主要集中在基于示例进行代码更新的问题，大多数现有的技术（例如，[8,39,52]）主要针对在同一代码库中的重复更新。这些技术通常假设更新示例作为输入提供。这意味着，这些技术主要关注如何从给定的示例中学习和应用代码更新，而这些示例通常是在同一代码库中找到的，也就是说，它们主要关注如何在同一应用程序的不同版本之间应用和复用代码更新。（给定弃用的API和更新的API，学习如何更新的）

方法

本文提出了AppEvolve，一种通过利用现有代码库自动化应用程序从其底层API的旧版本到新版本的演变的技术。它找到并分析已经经历了同样更新的现有应用程序的代码。该技术主要分为四个阶段：API使用分析、更新示例搜索、更新示例分析和API使用更新。

贡献

本文的主要贡献包括：

  • AppEvolve：这是一种新的技术，当应用程序的底层API发生变化时，可以自动更新应用程序。这种技术基于提取其他应用程序相似更新的现有示例。
  • 工具：实现了AppEvolve的工具，该工具是公开可用的，连同我们的实验数据和基础设施，可以在这个网址找到：https://sites.google.com/view/appevolve。
  • 实证评估：对AppEvolve进行了实证评估，应用于15个真实世界的应用程序，展示了我们的技术的有效性、效率和整体实用性。

TERMINOLOGY & MOTIVATING EXAMPLE

本文的动机示例涉及到Android API版本23的变化。原来的方法getAllNetworkInfo()，它返回设备支持的所有网络类型的连接信息，在Android API的版本23中被弃用，取而代之的是两个新的方法getAllNetworks()和getNetworkInfo(Network)。

文章给出了这个API变化的两个不同的更新，分别被称为updateV1和updateV2，这两个更新都来自于真实应用的实际更新。

在updateV1中，开发者引入了一个检查来确定应用程序正在运行的平台的版本。如果应用程序正在旧版本的平台上运行，代码会检查设备是否连接到网络，这与旧版本的代码执行的方式相似。相反，如果应用程序正在新版本的平台上运行，代码会调用新API的getAllNetworks()，并迭代Network对象的数组，以确定设备是否连接到网络。为了确定连接状态，开发者从Network对象中调用getNetworkInfo来获取网络信息。

updateV2与updateV1类似，因为两次更新都引入了一个检查来确定应用程序正在运行的平台的版本，并在条件的一个分支中调用新API的getAllNetworks和getNetworkInfo，在另一个分支中调用getAllNetworkInfo。然而，这两次更新也有一些差异。updateV1通过将调用getState的结果与CONNECTED进行比较来检查NetworkInfo对象的连接状态，而updateV2则使用isConnected。此外，updateV2还记录了正在连接的网络类型的信息，并进行了与API使用无关的更改。

这个例子突出了使用更新示例来更新API使用的潜力，同时也展示了自动执行此操作所涉及的一些挑战。这个例子显示，不同的更新可以有共性，但也有差异。

TECHNIQUE

AppEvolve，这是一种自动执行基于更新示例的API使用更新的技术。这种技术特别针对与Android平台API的更改相关的更新。

AppEvolve的基本概念是通过利用其他应用的开发者如何在他们的应用中更新相同的API使用情况，来更新目标Android应用中的API使用情况。

AppEvolve主要分为四个阶段进行操作：

  1. API使用分析阶段：该技术分析应用的源代码，以识别需要更改的API使用情况，并将此信息存储在API使用报告中。
  2. 更新示例搜索阶段：此阶段使用API使用分析阶段的信息和AU更改规范，来在现有的代码库中搜索API使用更新。
  3. 更新示例分析阶段：此阶段处理识别到的示例，以将它们概括化，计算它们的共同核心，并根据与共同核心的接近程度对它们进行排名。在此过程中，示例被转换为通用的更新补丁。
  4. API使用更新阶段：此阶段使用通用的更新补丁来更改AppEvolve的第一阶段报告的API使用位置，并使用差异测试进行验证。

该技术的最终输出是一个演化的目标应用和一个API使用更新报告，记录了应用中的更改。

3.1 API-Usage Analysis

在API使用分析阶段，输入是目标应用（TA）的源代码和一组API使用更改（AUC1，…，AUCn），输出是一个API使用报告，报告中包含了需要更新的旧API使用在TA中的位置。每个API使用更改都是旧API和新API版本之间方法的映射。

AppEvolve通过检查旧API使用是否可以在新API版本上运行来识别需要更新的旧API使用。如果应用可以在多个API版本上运行，那么它的一些代码可能会被程序化地阻止在API的某些版本上运行。

该技术通过静态分析TA来计算这些信息。首先，AppEvolve计算出TA中的语句可以在哪些API版本（SAPI）上执行，这是通过利用相关工作中定义的一种过程间数据流分析来实现的。其次，该技术执行一种过程内分析，以识别旧API使用，并检查其中的方法调用是否可以在新API版本上执行（基于SAPI）。如果是这样，那么这些API使用就需要更新，所以AppEvolve将它们连同相应API调用的位置一起存储在API使用报告中。

3.2 Update Examples Search

“更新示例搜索”阶段的目标是找出每个在前一阶段报告的旧API用法的更新示例。这是通过分析其他应用的版本控制历史来查看它们如何更新相应的API用法来实现的。这个阶段的输出是一组针对目标应用中需要更新的每个旧API用法的更新示例。

这个阶段使用的算法接收旧API用法（AU）、对应的新API用法（AU’）以及一个代码托管基础设施（chi）的位置作为输入，其中其他应用的代码库可以公开访问。算法的输出是一组针对AU的更新示例（uesAU）。

算法首先通过在chi中使用AU’的文件中执行文本搜索来识别可能包含示例的代码库。这个搜索基于从AU’中提取的一组关键词和在chi的文件内容上构建的索引。这一步使得AppEvolve能够有效地考虑大型代码库，并快速丢弃无关的代码。

然后，算法处理每个由搜索结果产生的文件，以识别其历史中是否包含从AU到AU’的更新。它通过比较文件的每个版本与其前一个版本来实现这一点，从最近的版本开始。算法计算这两个版本之间的差异，并从旧版本中提取被移除的行，以及在新版本中添加的行。

算法检查旧版本和新版本之间是否存在从AU到AU’的更新。具体来说，它检查旧版本中是否没有使用AU’，并搜索在旧版本中使用AU的方法和在新版本中添加使用AU和AU’的新行的对应方法（在检查API版本的条件的不同分支中）。

对于找到的每个更新示例，算法存储以下信息：代码库及其版本历史信息，更新前后的文件版本，包含AU和AU’的旧版本和新版本的方法签名，以及与此更新相关的API用法（AU和AU’）。最后，算法返回找到的更新示例列表。

3.3 Update Examples Analysis

Generic Update Patch Generation.

  1. 创建一个空的通用更新补丁列表：算法首先创建一个空的通用更新补丁列表。
  2. 处理每个示例：处理与API使用更改相关的每个更新示例。算法识别出一系列编辑操作，这些操作将代码的旧版本中的方法转换为新版本中的方法。这些编辑被编码为在为方法构建的抽象语法树（AST）上执行的树操作。
  3. 识别相关的编辑：算法使用内部过程依赖性分析来识别与API使用更改相关的编辑。它选择那些影响或受到来自旧API使用和新API使用的方法调用中涉及的值影响的编辑。
  4. 丢弃无关的编辑：在分析结束时，所有与更新无关的编辑都被视为无关并被丢弃。
  5. 编辑的概括：到目前为止识别的编辑依赖于从中计算出的特定示例。算法将它们概括，以便可以用它们来更新目标应用程序中的方法的API使用。每个编辑都被翻译成一个通用编辑，它包括原始编辑、受编辑影响的语句的位置，以及语句的抽象。
  6. 提取上下文变量：算法识别出在新版本的AST中的语句使用但在这些语句中没有定义的变量，这些语句受到编辑的影响。这些被称为上下文变量，因为它们为更新提供了上下文。
  7. 创建通用更新补丁：上下文变量和通用编辑定义了通用更新补丁的内容，算法将其添加到补丁列表中。

Generic Update Patch Prioritization.

一旦所有的更新示例被翻译成通用更新补丁，算法就会根据它们与一个共享核心的关系来对它们进行排序。这个共享核心被定义为所有补丁共享的最长子序列编辑，通过解决一个多个最长共享子序列问题（MLCS）来计算。

为了处理在不同变量上操作的补丁，该技术在将编辑加入到MLCS问题之前使用了编辑的抽象。这个抽象是一个字符串，包含了编辑的类型和编辑涉及的语句的抽象。

在计算核心之后，算法计算每个补丁的”核心接近度”，这表示一个补丁与核心有多近的关系。这是通过将核心中的编辑数量除以补丁中的编辑数量来计算的。然后，补丁会根据这个值进行排序。

给出的例子解释了，updateV1和updateV2的补丁的核心接近度分别为0.77和0.67。这表明updateV1的补丁与核心的关系更近，尽管它没有最大值（1），这是因为使用了不同的语句，这影响了核心的计算。

为了避免丢弃执行更新所必需的语句，决定对补丁进行排名，而不是合并它们。算法的最终输出是补丁的有序列表（gupsAU change）。

3.4 API-Usage Update

该算法以目标应用、需要更新的API使用、API使用在目标应用中的位置、当前API变化的通用更新补丁，以及用于验证更新的测试套件作为输入。其输出为更新的目标应用和一个API使用更新报告，在其中为开发者记录了特定API变化的变化。

算法从与更新核心最接近的补丁开始处理每个补丁。对于给定的补丁，算法首先确定补丁是否适用于目标应用中的API使用。补丁的适用性取决于两个因素：一是与补丁相关联的上下文变量必须在应用补丁的程序点中可用；二是必须能够成功地将通用编辑（全部）应用于使用API的目标应用的方法。

在确定通用更新补丁的适用性时，算法首先根据API使用的位置构建包含API使用的方法的AST，然后遍历AST中的语句节点以找到应用通用更新补丁的点。算法通过计算语句处的变量范围，并确定是否存在从范围变量到上下文变量的映射来做到这一点。如果找到成功的映射，算法将根据映射替换其上下文变量，然后在AST中执行补丁的编辑。

如果所有的编辑操作都成功，那么Apply-Update-Patch函数将返回一个新的AST，该AST编码了API的更新。此时，算法认为补丁适用于目标应用，并更新AST的代码，从而生成更新的应用。然后，它使用差分测试来验证更新的应用。

如果存在可用的测试套件并且能够成功运行API，那么就使用这个测试套件。否则，如果没有测试套件，就尝试使用随机输入生成器在API的旧版本上运行目标应用来创建一个。如果生成的测试套件能够成功运行API，那么就将其作为测试套件使用。否则，将空的测试套件传递给算法，此时算法将不验证更新，并将代码更改记录在API使用更新报告中，标记为可应用但未验证。

如果测试套件真实存在，那么就使用它来验证更新。在新旧API版本上运行更新后的应用来执行测试套件，以检查向后兼容性。如果没有失败，那么更新就被认为是有效的，将代码更改记录在API使用更新报告中，并标记为已验证，然后移动到API使用报告中的下一个API使用。否则，算法将持续验证更新，直到能够验证一个或者已经分析了所有可能的更新。

在处理了API使用报告中的所有API使用后，此阶段的输出是一个演进的目标应用和一个API使用更新报告，其中记录了验证和适用的更新。

Evaluation

作者设计了以下三个研究问题：

  • RQ1：AppEvolve能否成功地更新真实应用中的API使用？
  • RQ2：对于AppEvolve识别的更新示例，AppEvolve和LASE在有效性方面如何比较？
  • RQ3：运行AppEvolve的成本是多少？

Benchmark：作者使用了15个来自F-droid仓库的真实世界应用作为基准测试。这些应用被选中是因为它们基于最新支持的API版本进行分类。作者选择了三组，每组五个应用，每组应用具有相同的最新支持的API版本：22、23和25。对于每个API版本，作者通过研究相应的API文档手动生成了一个API更改规范。此外，确保每个应用包含至少一个API使用，该使用与其他应用中的使用不同，并且必须在后续的API版本中进行更新。

Metrics：作者使用了几个指标来评估AppEvolve的性能，包括：

  • 成功更新API使用的数量和比例
  • AppEvolve和LASE在有效性方面的比较
  • 运行AppEvolve的时间成本

Baselines：作者选择了LASE作为基线进行比较，因为LASE也从示例中提取编辑脚本，与AppEvolve最相关，处理Java程序，并且公开可用。

结果：

  • 对于RQ1，AppEvolve能够成功地更新85%的API使用，并且能够自动验证68%的这些更新。
  • 对于RQ2，当用于自动执行API更改的应用更新时，AppEvolve比LASE更有效。
  • 对于RQ3，AppEvolve的运行时间主要由更新示例搜索阶段决定，平均需要10小时27分钟。但是，这个阶段可以通过离线索引仓库来加速，并且可以并行运行和在夜间运行（并且结果可以在开发者之间共享）。所有其他阶段都可以相当有效地运行。
