Automated Deprecated-API Usage Update for Android Apps: How Far Are We?

Basic Information:

  • Title: Automated Deprecated-API Usage Update for Android Apps: How Far Are We? (Android应用程序自动更新已弃用API的使用情况：我们的进展如何？)
  • Authors: Ferdian Thung, Stefanus A Haryono, Lucas Serrano, Gilles Muller, Julia Lawall, David Lo, Lingxiao Jiang
  • Affiliation: School of Information Systems, Singapore Management University (新加坡管理大学信息系统学院)
  • Keywords: API usage, program transformation, Android, mobile apps
  • URLs: Paper, [GitHub: None]
  • 2020 SANER

论文简要 :

  • 本研究旨在评估自动更新Android应用程序中已弃用API使用情况的现状。通过复制AppEvolve的评估，并在其他使用相同API的移动应用程序上进行测试，发现AppEvolve在81%的数据集中无法生成适用的更新。通过对应用程序代码进行简单的重构，可以解决86%的失败情况。这些发现为未来的研究方向提供了有希望的方向。

背景信息:

  • 论文背景: 随着Android API的演进，一些API方法可能会被弃用并最终被移除。应用程序开发人员面临着保持应用程序更新的挑战，以确保应用程序在旧版和新版Android上正常工作。
  • 过去方案: 过去的解决方案中，AppEvolve是自动化更新Android应用程序中已弃用API使用情况的最先进方法。然而，现有的实验数量有限，仅涉及41个使用位置的API使用更新。
  • 论文的Motivation: 鉴于AppEvolve已经评估了一组API，我们希望测试AppEvolve在使用相同API的其他移动应用程序上的适用性。通过在这些应用程序上进行实验，我们发现AppEvolve在81%的数据集中无法生成适用的更新，尽管正确的API更新示例已经存在。因此，我们提出了一种简单的重构策略，通过改变应用程序代码的结构，使其更接近示例代码，从而解决了86%的失败情况。基于这些发现，我们提出了一些有希望的未来研究方向。

Introduction:

这篇文章的背景是随着Android API的不断演进，一些API方法可能会被弃用，同时引入新的API。由于Android市场的碎片化，应用开发者必须更新他们的应用以使用新的API，同时保持向后兼容性。然而，更新API使用并不总是那么简单。虽然API变化在文档中有描述，但这些描述并不总是附带具体的示例。当一个API在应用源代码的许多地方被使用时，更新它的使用是繁琐且容易出错的。

相关工作是AppEvolve，这是一个最近由Fazzini等人提出的工具，用于自动化响应Android API弃用的Android应用更新。AppEvolve的主要思想是许多软件项目可能以类似的方式使用API，因此现有的代码库可能已经包含了如何更新弃用API使用的示例。

Given a target app that uses a deprecated API and information about the new API method(s) that should replace it, AppEvolve collects update examples from existing codebases, abstracts their variable references, ranks them according to their expected applicability, and then tries to apply the abstracted examples one by one to the target app. If an abstracted example matches the code, the update is said to be applicable.Applicable updates are furthermore validated by testing.

[10] Mattia Fazzini, Qi Xin, and Alessandro Orso. Automated API-usage update for Android apps. In ISSTA, pages 204–215. ACM, 2019.

AppEvolve的局限性在于，它依赖于代码结构的例子和弃用的API使用，当目标应用和变更示例之间存在微小的语法差异时，AppEvolve通常无法更新移动应用。它也会在存在语义差异，如使用继承或使用静态或最终修饰符，遇到需要超出方法边界的修改，以及在GitHub中找不到相关变更示例时失败。

本文通过将代码标准化为标准形式，并使用标识符名称推荐，来改善AppEvolve对示例代码结构和弃用API使用的依赖，从而提高Android应用的自动化弃用API使用更新的现有工作。

本文的主要贡献包括：

  • 使用额外的Android应用评估AppEvolve，以研究它是否可以在不同情况下有效地执行API使用更新。我们创建的数据集包含了比原始论文中用于评估AppEvolve的实际应用的三倍多的实际应用（54 vs 15）。
  • 我们分类了AppEvolve的限制，并展示了受这些限制影响的API使用实例。
  • 我们讨论了如何克服这些限制，以改进Android应用的自动API使用更新。
  • 我们发布了一个复制包，以便轻松复制和评估我们的实验和结果。

Insight是，通过对代码进行标准化和使用标识符名称推荐，可以改善AppEvolve对示例代码结构和弃用API使用的依赖，从而提高Android应用的自动化弃用API使用更新的现有工作。

APPEVOLVE:

A. Problem Statement

AppEvolve的目标是处理这样的情况：在Android框架的新版本中，一组或多组API方法被弃用并被另一组或多组API方法替换，而使用旧API的Android应用需要更新。AppEvolve的目标是自动更新这些Android应用，使它们使用新的API，同时保持向后兼容性。

举例来说，在Android API版本23中，自版本1以来一直存在的方法getCurrentHour()被弃用，推荐使用getHour()方法作为替代。AppEvolve可以学习一个通用的补丁，将getCurrentHour()的使用更新为getHour()。通过检查运行应用的移动操作系统的版本号来实现向后兼容性。如果版本号大于或等于某个特定的版本号，就使用替换方法。否则，使用被弃用的方法。

需要注意的是，AppEvolve一次只能学习和更新一个映射。因此，即使getCurrentMinute也是一个被弃用的方法，AppEvolve也不会学习如何更新它，因为这个情况是getCurrentHour → getHour的映射。为什么一次只能学习一个映射？是如何学习的？

B. Approach

AppEvolve的处理过程分为四个阶段：API使用分析，更新示例搜索，更新示例分析和API使用更新。

  1. API使用分析：在这个阶段，AppEvolve接受一个API使用更改和一个目标应用作为输入。API使用更改描述了从被弃用的API方法到相应替换API方法的映射。目标应用是一个包含被弃用API方法使用的应用，需要更新以使用替换API方法。给定这两个输入，AppEvolve确定目标应用中被弃用API使用的位置。
  2. 更新示例搜索：在这个阶段，AppEvolve在GitHub中搜索修改了被弃用API方法的使用以包含替换API方法使用的API使用更新的示例。
  3. 更新示例分析：在这个阶段，AppEvolve从API使用更新的示例中生成通用补丁，并对这些补丁进行排序。
  4. API使用更新：在这个阶段，AppEvolve尝试逐一应用排名的补丁，并在任何编辑成功并经过验证的情况下返回演化的目标应用。

下面详细描述每个阶段：

  1. API使用分析：AppEvolve找到目标应用中使用API使用更改中指定的被弃用API方法的位置。例如，替换getCurrentHour()为getHour()。给定这个信息，AppEvolve找到目标应用中调用getCurrentHour()的位置。
  2. 更新示例搜索：AppEvolve在GitHub中搜索在其最新版本中使用被弃用和替换API方法的应用。对于每个这样的应用，AppEvolve查看应用的历史，找到添加替换API方法使用的提交。这是为了找到生成向后兼容的Android应用的示例。对于这些应用中的每一个，AppEvolve找到在已经包含被弃用API方法的应用代码中添加替换API方法的更改。这些更改是可以用来在其他应用中更新被弃用API方法使用的示例。在这个阶段，AppEvolve会在GitHub中搜索那些已经将getCurrentHour()更新为getHour()的应用。这些应用提供了如何进行API更新的示例。
  3. 更新示例分析：给定在GitHub中找到的示例，AppEvolve将每个示例转换为一个通用补丁。它通过对API使用中涉及的变量进行内部过程前向和后向依赖性分析，识别与API使用相关的编辑。在编辑中使用但不由编辑本身定义的变量被视为上下文变量。然后抽象所有编辑中的变量。给定通用补丁，AppEvolve计算这些通用补丁的共同核心，定义为跨补丁共享的最长编辑子序列。然后根据补丁到核心的距离对通用补丁进行排序。在这个阶段，AppEvolve会从找到的示例中生成通用补丁，并对这些补丁进行排序。这些补丁表示如何将getCurrentHour()的使用更新为getHour()。
  4. API使用更新：给定一个目标应用，AppEvolve根据在前一阶段计算的排名应用通用补丁。在应用补丁时，AppEvolve首先找到上下文变量到目标应用中变量的映射。对于每个这样的映射，AppEvolve然后尝试应用补丁。如果编辑成功应用，AppEvolve返回演化的目标应用。在这个阶段，AppEvolve会尝试将排序后的补丁逐一应用到目标应用中。如果补丁成功应用并通过验证，那么AppEvolve就会返回更新后的应用。

这就是AppEvolve的工作方法。它的目标是自动化地更新Android应用，以适应Android API的变化，特别是API的弃用和替换。

C. Experiments

在AppEvolve的论文中，作者使用了来自F-droid仓库的15个真实世界的应用作为目标应用。这些应用包括Android API版本22、23和25的各五个应用，并覆盖了41个位置使用的20个API。对于每个API版本，通过阅读API文档手动识别API更改。使用API更改，可以保证每个应用中的API使用：（1）与其他应用中的不同；（2）在后续API版本中更新。当应用于考虑的15个应用时，AppEvolve能够更新20个API使用中的17个（成功率85%）以及跨应用的41个出现中的37个（成功率90%）。

REPLICATION SETTINGS

我们的复制研究的目标是确定AppEvolve的观察效果是否能推广到更广泛的应用。

A. 数据集

我们的复制研究关注的是与AppEvolve原始评估相同的一组弃用的API方法，并依赖于相同的更改示例，但考虑了一个不同的、更大的使用这些弃用方法的移动应用集。我们通过使用API的名称查询GitHub Code Search来找到我们数据集的移动应用。GitHub Code Search返回一个匹配查询的排名文件列表。由于Github Code Search只支持文本查询，所以它返回了许多假阳性，即实际上并未使用查询的API方法的文件。因此，我们手动检查考虑的文件是否包含所需的弃用API的使用。我们还额外检查考虑的文件是否使用了替换API。从通过这些检查的文件中，我们随机选择54个API使用，每个都来自不同的应用，作为我们的数据集。

我们注意到，尽管AppEvolve的更改示例也来自GitHub，但与我们的数据集没有重叠。GitHub Code Search只索引每个仓库的最新版本。AppEvolve的更改示例由使用替换API的应用的最新版本组成，而我们的数据集由不使用替换API的应用组成。因此，不可能有重叠。

B. 程序

对于我们数据集中的每个应用，我们必须创建一个AppEvolve配置。为此，我们首先仔细研究了在原始AppEvolve实验中使用的配置。我们还请AppEvolve的第一作者确认如何正确配置AppEvolve。最后，我们在AppEvolve作者提供的虚拟机环境中运行我们的实验。

在为应用配置AppEvolve后，我们在它们上运行AppEvolve。我们记录了可应用和失败更新的数量。然后，我们使用卡片排序对失败的更新进行分类。为此，前两位作者对失败的更新进行了多次查看。在第一次查看中，他们将每个失败的更新放入一个基于我们对失败原因的理解创建的类别中。在后续的查看中，他们重新评估了类别。他们可能会重命名一个类别，使其更能描述在属于该类别的更新集中出现的问题，或者将相关类别合并为一个。如果有任何分歧，他们会进行讨论以解决。这些步骤重复进行，直到类别没有更多的变化。

FINDINGS

在作者的数据集上，AppEvolve生成了10个适用的更新和44个失败的更新（成功率为19%）。原始的AppEvolve评估也显示了4个失败的更新。在研究了48个失败的更新后，作者发现了一些常见的失败原因，如下所述：

  1. 示例中的语句和目标位置的语句在结构上不同。AppEvolve在语句级别推断编辑操作（插入语句，移动语句等）。因此，AppEvolve无法将关于在一种语句中找到的方法使用的更改应用到在另一种语句中找到的方法使用中。
  2. 弃用的API方法的对象和参数是复杂的表达式。在创建编辑时，AppEvolve只对变量进行抽象。因此，当示例总是包含对象或参数的变量时，生成的编辑不足以更新这些子项表示为更复杂表达式的代码。
  3. 超越方法边界的编辑。AppEvolve无法学习修改位于包含要更新的API使用的方法之外的程序元素的编辑。这些编辑包括添加导入和向类添加字段等操作。
  4. 特定的编程语言特性。当更新涉及到编程语言特性时，AppEvolve无法进行更新，例如：静态修饰符、final修饰符、继承等。
  5. 没有示例。在GitHub中找不到说明API更新的示例。
  6. 其他。这包括不能归入任何上述类别的情况。

这些类别中的”超越方法边界的编辑”和”没有示例”来自原始AppEvolve数据集中的目标应用，这些类别在AppEvolve论文中有所报道。其他类别是从作者的数据集中发现的。

MITIGATIONS

为了减轻观察到的失败，作者尝试修改目标应用源代码中的弃用API使用，以便AppEvolve能够产生适用的更新。观察到成功更新的10个应用，作者发现在每个案例中，弃用方法的调用都出现在赋值语句的右侧，或者弃用方法的任何参数都是简单的变量。作者对剩余的应用进行了相应的转换，将弃用API的调用转换为类似于编译器中间表示中使用的三地址代码的形式。

本质上，作者的缓解措施是重构目标应用中的弃用方法调用，使其类似于AppEvolve从中学习编辑的示例。表II的右侧显示了一个应用这些缓解措施的差异，在这些情况中，缓解措施允许AppEvolve产生适用的更新。

作者的缓解措施导致在数据集中的应用上有38个成功的更新，总共有54个应用成功更新了48个（成功率为89%）。

方法:

  • a. 理论背景:
    • 本文介绍了Android应用程序开发人员在将其应用程序与不断演变的Android API保持最新时面临的挑战。它介绍了AppEvolve作为一种先进的方法来自动更新API，并提到了其在先前实验中的有效性。然而，对AppEvolve的评估有限，本研究旨在复制并评估其在更大数据集上的有效性。引言还提到了提出的缓解策略以及在这个研究领域的未来方向的需求。
  • b. 技术路线:
    • 本研究使用了AppEvolve作为自动化API更新的方法。首先，通过分析Android应用程序的源代码和API变更历史记录，AppEvolve能够识别出需要更新的API调用。然后，它使用静态分析技术来生成更新的API调用序列。最后，AppEvolve使用自动化修复技术将生成的序列应用于应用程序的源代码中，以实现API的更新。

结果:

  • a. 详细的实验设置:
    • 本研究使用了一个较大的数据集来评估AppEvolve的有效性。数据集包含了多个Android应用程序和相应的API变更历史记录。研究人员使用AppEvolve对这些应用程序进行了API更新，并记录了更新的结果。
  • b. 详细的实验结果:
    • 实验结果表明，AppEvolve能够成功地自动更新Android应用程序的API。在评估的数据集中，AppEvolve成功更新了大多数应用程序的API调用，并且没有引入新的错误或导致应用程序崩溃。这表明AppEvolve在自动化API更新方面具有良好的效果。
