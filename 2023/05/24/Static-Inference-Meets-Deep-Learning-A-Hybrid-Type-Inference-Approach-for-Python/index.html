DL-based方法的局限性：

（i）无法保证类型的正确性(TypeWriter提出的)，预测结果可能是多个类型候选项，且可能与类型规则相矛盾；最先进的DL模型Typilus也会产生大约10%的预测，它们不能通过类型检查器的测试。Typilus和Typewriter采用类型检查器来验证DL模型返回的所有类型组合，并删除那些包含错误类型的组合。然而，这些方法不能纠正错误的类型，而只是过滤掉它们。

（ii）对于低频类型（在数据集中出现频率较低的类型）的预测不准确，而低频类型在实际注释中仍然占有重要比例。例如，相较于出现频率超过10,000次的类型，对于出现频率少于100次的类型，Typilus的准确率下降超过50%。

作者如何克服局限性：

为了克服现有研究的限制，本文提出了HiTyper框架，它结合了静态类型推断和深度学习模型的推荐（静态+深度学习）。框架引入了一种新颖的表示形式，称为类型依赖图（TDG），用于记录函数中变量之间的类型依赖关系。基于TDG，将类型推断任务转化为填充问题，通过静态方法和深度学习模型共同填充变量的类型。

贡献：

  1. 提出了一种混合类型推断框架HiTyper，将静态推断与深度学习相结合，实现更准确的类型预测。
  2. 设计了一种创新的类型依赖图（TDG），用于严格维护不同变量之间的类型依赖关系。
  3. 解决了之前研究面临的一些挑战，并设计了一系列类型拒绝规则和类型校正算法来验证深度学习模型的预测结果。
  4. 在两个公开数据集上进行了广泛的实验证明，HiTyper相较于现有的基线模型和静态类型推断工具具有卓越的性能。

Motivation Example:

文中描述了三种推导过程的方法：静态推导、深度学习(DL)方法和静态+深度学习方法。

  1. 静态推导：根据表格1，可以发现静态类型推导技术无法推导出参数text的类型，因为该参数位于数据流的开头，没有任何赋值或定义。为了推导类型，一种常见的解决方案是使用程序间分析，并捕捉调用parse()函数的函数[52]。然而，追踪程序中的函数，特别是某些库中的函数，不总是可行的。对于返回值，通过分析数据流和变量之间的依赖关系，静态推导可以轻松识别出shape（第5行，第13行）和placeholders（第6行，第10行）构成返回值。它可以递归地分析这两个变量的类型，并最终输出返回值的准确类型。事实上，Pysonar2和Pytype都可以正确推导出返回值是包含列表和字典的元组。
  2. DL方法：DL模型Typilus [2]根据参数text传递的语义和上下文信息准确地预测出类型为str。这个案例说明了DL模型可以预测出比静态推导更多的类型。然而，Typilus无法推导出parse()的返回值的类型。当前的DL模型无法保持变量之间的严格类型依赖关系。因此，Typilus只能推导出返回值的类型为元组，但无法准确预测元组内部的类型。在为Typilus的预测结果添加类型检查器时，它的参数预测被保留，因为它不违反任何现有的类型推导规则。然而，对于返回值，Typilus在表格1中的第2个和第3个类型预测被拒绝，因为parse()的返回值显式包含了两个具有不同类型的元素。第1个预测也被拒绝，因为它包含了在返回值中不存在的类型Optional[text]。在这种情况下，模型没有为返回值产生任何候选类型。
  3. 静态+DL方法：对于给定的代码示例，我们发现当满足足够的静态约束或依赖关系时，静态推导优于DL模型，而当类型缺乏足够的静态约束时，DL模型更适用。给定代码，HiTyper首先生成其对应的TDG（类型依赖图），如图1所示，并尝试将TDG中的所有节点填充为相应的类型（”blank filling”）。对于参数text，HiTyper确定无法通过静态推导来推断类型（因为参数text没有任何输入边），并向DL模型请求推荐。但是，HiTyper不直接将DL的预测结果输出为最终的类型分配。相反，HiTyper通过验证预测的正确性，只有在不违反任何类型推导规则的情况下才接受结果。当预测返回值时，HiTyper基于TDG捕捉其类型依赖关系（它与两个输入节点相连），并直接利用静态推导推断类型。对于这种情况，不需要DL的预测结果，从而大大避免了错误类型的引入。

综上所述，静态推导、DL方法和静态+DL方法是三种用于推导过程的不同方法。静态推导适用于具有足够静态约束的类型推导，DL方法可以预测更多的类型，而静态+DL方法结合了两者的优势，在满足静态约束不足的情况下利用DL的推荐，并通过静态推导验证结果的正确性。

Overview

HiTyper是一个接受Python源代码文件作为输入，并输出记录类型分配结果的JSON文件的工具。它的总体框架如图2所示，并包括三个主要组件：类型依赖图生成、静态类型推导和神经网络类型预测。静态类型推导组件包括两个主要步骤，即前向类型推导和后向类型拒绝。

类型依赖图（TDG）生成：首先，给定一个Python源代码文件，HiTyper为每个函数生成TDG，并识别所有导入的用户定义类型（第3.3节）。TDG将每个变量的声明语句和表达式转换为节点，并维护它们之间的类型依赖关系，以便静态推导和DL模型可以共同将类型填充到其中。

静态类型推导 - 前向类型推导。为了保持预测结果的正确性，HiTyper主要通过静态推导来推断类型。给定一个TDG，HiTyper通过遍历图并实施保存在每个表达式节点中的类型推导规则来进行前向类型推导（第3.4节）。然而，由于静态推导的限制，在大多数情况下，HiTyper只能推断部分类型槽，即在图2中部分推断的TDG中表示为黑色实心节点的变量；而空白节点表示缺乏足够静态约束且未解决的类型槽。为了增强HiTyper的推断能力，我们向DL模型请求推荐。

神经网络类型推荐。通过热类型槽查找器，HiTyper确定一部分空白节点作为热类型槽，标记为图2中的红色节点，并从DL模型获取推荐（第3.5节）。HiTyper还使用基于相似度的类型校正算法来补充用户定义类型的预测，这些类型是稀有类型的主要来源。神经网络类型预测组件推荐的类型被填充到图中，得到推荐的TDG。

静态类型推导 - 后向类型拒绝。HiTyper利用类型拒绝规则验证热类型槽中的神经网络预测（第3.4节）。然后，它遍历整个TDG，将拒绝的预测从输出节点传输到输入节点，以便验证TDG中的所有节点。最后，HiTyper再次调用前向类型推导，根据经过验证的推荐推断新的类型。

前向类型推导和后向类型拒绝之间的交互可以迭代，直到TDG达到一个稳定状态，即所有节点的类型不再改变。同时，静态推导和神经网络预测之间的迭代可以重复多次，直到推断出所有的类型槽，或达到最大迭代次数。这样可以确保HiTyper在静态推导和神经网络预测之间进行有效的交互，以最大程度地推断出准确的类型。

Type Dependency Graph Generation

HiTyper生成类型依赖图（TDG）的过程如下：

  1. 导入分析（Import Analysis）：在为每个输入函数建立TDG之前，HiTyper首先进行导入分析，提取所有用户定义的类型。它收集源代码中的所有类，并将其组成初始的用户定义类型集合。然后，对所有局部导入语句（如“from package import class”）进行分析，并将导入的类添加到用户定义类型集合中。对于全局导入语句（如“import package”），HiTyper定位该包的源代码，并将源代码中的所有类和命名元组添加到用户定义类型集合中。对于每个导入的类，HiTyper解析外部源文件的位置，并检查该类中是否存在运算符重载方法。
  2. TDG生成（Type Dependency Graph Generation）：给定输入代码的AST（Abstract Syntax Tree）以及通过导入分析提取的所有用户定义类型，HiTyper根据算法1中的主要逻辑为每个函数创建TDG。HiTyper通过遍历整个AST来定位代码中的变量和表达式。对于每个AST节点，HiTyper使用Python的ast模块提供的ASTVisitor进行访问。根据Python中表达式节点的定义（如图4所示），HiTyper识别表达式并使用表达式栈记录每个访问的表达式节点。每当HiTyper识别到一个表达式节点时（第3行），它会在当前TDG中构建相同的节点，并将其推入表达式栈中。然后，HiTyper将递归访问表达式的操作数以捕获新的表达式节点，直到遇到变量节点（第12行），该节点是AST的叶节点。
  3. 符号节点和表达式节点的构建：HiTyper为每个AST中访问的标识符节点在TDG中创建一个符号节点，并维护一个变量映射表来记录每个变量的出现情况。根据AST已经指示的每个变量出现的上下文（读或写），HiTyper执行以下操作：
    • 如果变量的上下文是读取，则根据当前上下文在维护的变量映射表中获取该变量的最后一次出现。然后，它在符号节点的最后一次出现和当前变量的符号节点之间创建一条边。
    • 如果变量的上下文是写入，则从表达式栈中获取最后一个表达式的值，并在表达式节点和当前变量的符号节点之间创建一条边。
  4. 数据流分支和合并：类似于常规数据流分析，HiTyper还检查数据流在特定位置的分支和合并情况。

在TDG中，每个符号节点保存候选类型的列表，而每个表达式节点包含类型推测规则和类型拒绝规则。当HiTyper遍历TDG时，这些规则将被激活以生成新的类型。因此，类型可以从参数流向返回值。通过遍历，HiTyper获取每个符号节点的类型，并输出类型赋值结果。

总结起来，TDG的生成过程包括以下步骤：

  1. 进行导入分析，提取所有用户定义的类型。
  2. 遍历输入代码的AST，识别变量和表达式节点。
  3. 构建符号节点和表达式节点，并在TDG中建立相应的节点。
  4. 根据变量的上下文（读或写），建立节点之间的边，表示类型之间的依赖关系。
  5. 检查数据流的分支和合并情况，更新TDG中的节点和边。
  6. 应用类型推断规则和类型拒绝规则，从参数到返回值的方向推导出节点的类型。
  7. 遍历TDG，获取每个符号节点的类型，并输出类型赋值结果。

通过这个过程，HiTyper能够生成描述程序中不同变量之间类型依赖关系的TDG，为后续的静态类型推断和神经网络类型预测提供基础。

Static Type Inference

HiTyper是一个用于进行静态类型推导的工具，它使用了正向类型推导和反向类型拒绝的策略。以下是论文中描述的关键概念和步骤：

  1. 正向类型推导（Forward Type Inference）：HiTyper从不具有输入节点的节点开始进行正向类型推导。它逐步访问图中的所有节点，并在满足前提条件的情况下激活相应的类型推导规则。前提条件包括输入节点的类型已经完全推导出来。这个过程类似于传统的静态类型推导技术。正向类型推导的目标是推导出当前表达式节点的结果类型。（这个感觉和pysonar类似）
  2. 反向类型拒绝（Backward Type Rejection）：在一个表达式中，输入类型必须满足两个约束条件：（1）它必须是进行特定操作的有效类型，（2）它必须与其他输入类型存在有效的关系。HiTyper会拒绝违反这两个约束条件的输入类型。首先，它检查类型是否对于当前表达式是有效的。其次，它检查输入类型之间的关系是否有效。一些操作还要求输入满足特定的关系条件。HiTyper根据正向类型推导规则和类型约束来验证和拒绝输入类型。拒绝的输入类型可能会影响先前操作的输入类型。为了彻底消除错误类型的影响，HiTyper还会根据正向类型推导规则拒绝导致错误类型的输入类型。这个过程是对图的反向遍历。(这个感觉和类型检查器类似)
  3. 类型推导和拒绝规则（Type Inference and Rejection Rules）：论文中的图示（Fig. 5）列出了HiTyper中使用的所有类型推导和拒绝规则。每个规则包含前提条件（上方内容）和结论（下方内容）。规则的形式是𝜋 𝑒 : 𝜃，其中𝜋是上下文，包括将类型分配给表达式模式的列表；𝑒是表达式；𝜃是类型。规则被称为类型判断或类型分配。规则的结论部分包括当前表达式节点的结果类型和输入节点的验证类型。类型推导规则根据前提条件推断出当前表达式的结果类型，而类型拒绝规则用于验证和拒绝输入节点的类型。
  4. 正确性（Correctness）：HiTyper的类型分配始终保持正确性。如果静态推导可以成功处理程序，则HiTyper不需要调用深度学习模型提供类型推荐。在这种情况下，完全基于推导规则（Fig. 5）生成的类型分配是准确的，因为它们是从Python官方实现CPython [10]收集而来的。如果静态推导无法完全推导出程序，并且需要调用深度学习模型提供类型推荐（第3.5节），HiTyper会使用类型拒绝规则验证推荐结果，然后再次调用类型推导规则来推导剩余的类型。在这种情况下，拒绝规则会彻底消除错误推荐的影响，最终结果仍然是由静态推导得出的。因此，HiTyper始终保持类型的正确性。

HiTyper使用正向类型推导和反向类型拒绝的策略来进行静态类型推导。正向类型推导根据推导规则逐步推导出表达式节点的结果类型，而反向类型拒绝根据推导规则和类型约束验证和拒绝输入节点的类型。这种组合策略确保了类型分配的正确性。

Neural Type Recommendation

对于用户定义的类型，一些DL模型将其视为Out-of-Vocabulary（OOV）标记，并不预测类型[18,39]

[18] Vincent J. Hellendoorn, Christian Bird, Earl T. Barr, and Miltiadis Allamanis.Deep learning type inference. In Proceedings of the 2018 26th ACM Joint Meeting**on European Software Engineering Conference and Symposium on the Foundations of Software Engineering*, ESEC/FSE 2018, page 152–162, New York, NY, USA, 2018.Association for Computing Machinery.

[39] Michael Pradel, Georgios Gousios, Jason Liu, and Satish Chandra. *TypeWriter:*Neural Type Prediction with Search-Based Validation, page 209–220. Associationfor Computing Machinery, New York, NY, USA, 2020.

而其他模型则直接从训练集中复制用户定义的类型，但无法预测在训练集中从未出现的类型。[2,45]

[2] Miltiadis Allamanis, Earl T. Barr, Soline Ducousso, and Zheng Gao. Typilus:Neural type hints. In Proceedings of the 41st ACM SIGPLAN Conference on Programming Language Design and Implementation, PLDI 2020, page 91–105, NewYork, NY, USA, 2020. Association for Computing Machinery.

[45] Pytype. https://github.com/google/pytype.

作者选择Typilus和Type4Py作为HiTyper接受类型推荐的神经网络模型。

关于处理自定义类型：

在相似度校正算法中，HiTyper首先检查推荐的类型是否属于内置类型，如果不是，则检查该类型是否在通过导入分析收集的用户定义类型集合中出现。如果检查结果为False，则表示该类型是明显不正确的，需要进行校正。对于这些不正确的用户定义类型，HiTyper使用Word2Vec将两个类型和变量名嵌入为词嵌入，并计算余弦距离作为两个类型的相似度。最后，HiTyper选择与用户定义类型集合中最相似的候选类型来填充用户定义类型。

让我们通过一个例子来说明相似度校正算法的工作原理。

假设在代码中有一个用户定义的类型为MyType的变量，但是静态类型推导无法确定该变量的准确类型。DL模型进行类型推荐时，给出了一个不正确的推荐类型MType。在这种情况下，HiTyper使用相似度校正算法来纠正该推荐类型。

首先，HiTyper通过导入分析收集了一组用户定义的类型，其中包括MyType。接下来，HiTyper使用Word2Vec将MyType、MType和变量名嵌入为词嵌入向量，并计算它们之间的余弦相似度。

假设经过计算，得到以下相似度值：

  • 相似度(MyType, MType) = 0.85
  • 相似度(MyType, 其他用户定义类型1) = 0.70
  • 相似度(MyType, 其他用户定义类型2) = 0.60

根据相似度值，HiTyper选择与MyType最相似的候选类型来填充用户定义类型。在这种情况下，MyType与MType的相似度最高，因此HiTyper将使用MyType作为纠正后的类型，将其填充到相应的变量中。

通过相似度校正算法，HiTyper能够利用用户定义类型之间的相似性来修正错误的类型推荐，提高推荐结果的准确性。

Evaluation:

在该部分，作者回答了以下研究问题：

  • RQ1：与Baselines相比，HiTyper的效果如何？
  • RQ2：HiTyper能否很好地预测罕见类型？
  • RQ3：HiTyper中的静态类型推断组件的性能如何？

Benchmark

数据集：作者在评估中使用了2个Python数据集。一个是由Allamanis等人发布的Typilus数据集[2]，另一个是由Mir等人发布的ManyTypes4Py数据集[34]。关于测试集中不同类型的数量和更详细的统计信息，请参考表3和第2节。

Metrics

评估指标：作者选择了两个评估指标：Exact Match（完全匹配）和Match to Parametric（参数匹配）。这两个指标分别计算以下结果的比例：

1）Exact Match：与人工注释完全匹配；

2）Match to Parametric：在忽略所有类型参数的情况下满足完全匹配。例如，List[int]和List[str]在此指标下被视为匹配。

Baselines

基准方法：为了验证HiTyper的有效性，作者选择了五种基准方法进行比较：

1）一个简单的基准方法。它代表一种基本的数据驱动方法。该基准方法遵循[39]的工作，通过从最常见的十种类型的分布中进行采样来进行预测。【39】 TypeWriter

2）Pytype [45]和Pyre Infer [40]。它们是来自Google和Facebook的两种流行的Python静态类型推断工具。

3）Typilus [2]和Type4Py [35]。Typilus是一个利用代码结构信息的图模型。Type4Py是一个使用类型聚类来预测类型的层次神经网络。

Results

对RQ1的回答： HiTyper在总体类型推断性能上显示出了很大的改进（11%∼15%），而最显著的改进是在返回值推断上（22%∼39%）。

对RQ2的回答： HiTyper利用了静态类型推理组件，实现了>30%的提升，从而大大缓解了DL模型所面临的罕见类型的预测问题。

对RQ3的回答：只考虑静态推理部分，HiTyper仍然优于当前的静态类型推理工具，通过推断2×∼3×更多的变量，具有更高的准确性。

Conclusion:

  • a. Significance of the work:
    • 本文提出的HiTyper方法对于动态类型编程语言中实现高精度的类型推断非常重要。
  • b. Innovation, performance, and workload:
    • 本文提出了一种新的混合类型推断方法，并通过实验显示其比传统的静态类型推断工具和深度学习模型具有更好的性能和工作量。
  • c. Research conclusions (list points):
    • HiTyper采用类型依赖图和静态推断规则来推断变量的类型依赖关系，同时使用深度学习模型进行推断。该方法具有较高的准确性和有效性，能够纠正存在的错误注释，适用于实际生产。
